# Harmony Palette — CVD Evaluation

> An **original analysis** (not a tool benchmark): we took Evil Martians' [Harmony palette](https://github.com/evilmartians/harmony) — marketed as a perceptually-uniform, APCA-tuned, accessible Tailwind replacement — and evaluated it against the color-vision-deficiency findings from [COLOR_AND_CONTRAST_RESEARCH.md](./COLOR_AND_CONTRAST_RESEARCH.md). The hypothesis (from the user, who hit this in practice): Harmony's hues become indistinguishable under CVD, worst under monochromacy. **Confirmed, decisively, with a clear mechanical root cause.** 2026-06-20.

This matters because Harmony is the closest thing to "what a naïve okLCH+APCA palette generator produces," and the failure is exactly the gap our tool should close (the ΔE-under-CVD feature from the color-&-contrast research).

---

## 1. What Harmony is

From its README: _"Accessible UI Color Palette… Uses OKLCH and APCA for highly consistent color shades… Equal contrast within lightness groups… Mirrored contrast pairs… Tailwind compatibility."_ It ships **25 hues × 11 shades (50–950) = 275 swatches** as DTCG tokens; each stores its `$oklch`, `$srgbFallback`, `$isP3`, and notably **`$contrast`** (the APCA Lc it was tuned to) and `$contrastBg`. Hue set = the full Tailwind range (Red…Rose) plus grays (Slate/Gray/Zinc/Neutral/Stone/Sand/Olive/Mauve).

It was generated by [Harmonizer](https://github.com/evilmartians/harmonizer) — the EM tool we'd deferred studying; this evaluation effectively covers its _output_.

---

## 2. Method

For a given shade level, take the **17 chromatic hues** (excluding grays), then:

1. **Simulate** each swatch under **protanopia / deuteranopia / tritanopia** using the standard **Machado et al. (2009)** linear-RGB matrices (severity 1.0 — the same model Chrome DevTools and most simulators use), and under **monochromacy** via Rec.709 relative-luminance greyscale.
2. Compute **ΔE2000** (colorjs.io) between every hue pair in the simulated space.
3. Count pairs below distinguishability thresholds: **ΔE < 2** ≈ effectively the same color ("invisible" difference); **ΔE < 5** ≈ hard to tell apart ("weak"). (Recall ΔE = 1 is a just-noticeable difference.)

Scripts are reproducible; matrices and code are recorded in §6.

- **Normal vision: ~all distinguishable** (worst ΔE 5–8). Harmony looks great — until you test for CVD. This is _why_ the failure is easy to miss.

## 4. Root cause — the strength _is_ the weakness

The mechanism is not subtle. At shade 500, every chromatic hue is engineered to the **same APCA contrast**, which means the **same luminance**:

| Hue (shade 500) | stored APCA `$contrast` | OKLCH L        | Rec709 luminance Y |
| --------------- | ----------------------- | -------------- | ------------------ |
| Red             | −54.2                   | 73.2%          | 36.9%              |
| Yellow          | −54.3                   | 72.5%          | 37.9%              |
| Green           | −54.1                   | 71.9%          | 38.5%              |
| Cyan            | −54.3                   | 71.4%          | 38.7%              |
| Blue            | −54.1                   | 72.2%          | 38.0%              |
| Violet          | −54.2                   | 73.1%          | 37.7%              |
| Rose            | −54.3                   | 73.4%          | 37.0%              |
| _(all 17)_      | **−54.0 … −54.3**       | **71.3–73.4%** | **36.9–38.7%**     |

Spread across all 17 hues: APCA **0.3 Lc**, luminance **~1.8%**. They are, by design, the _same lightness_ — they differ **only in chroma and hue**.

CVD compresses the chromatic (red-green / blue-yellow) channels while largely preserving the luminance channel. Monochromacy removes the chromatic channels _entirely_. So:

> **Harmony's defining feature — "equal contrast within a lightness group" — equalizes luminance across hues. The moment a viewer's chromatic discrimination is reduced, the only remaining signal (luminance) is identical, and the hues collapse into one another.** The more uniform the contrast, the more complete the collapse.

This is the concrete, quantified version of the color-&-contrast guide's warning: _uniform-contrast/perceptual-space palettes can "falsely identify safe colors"_ because they optimize luminance consistency, not cross-hue **distinguishability**. The two goals are in direct tension along the luminance axis.

## 5. Implications for our tool

This is the strongest possible case for the **ΔE-under-CVD "distinguishability matrix"** feature (from [COLOR_AND_CONTRAST_RESEARCH.md](./COLOR_AND_CONTRAST_RESEARCH.md) §13):

1. **Ship the CVD ΔE check as a first-class accessibility view**, not a footnote. For a _categorical_ set of hues (status colors, chart series, brand accents), simulate each through protan/deutan/tritan/mono and surface the ΔE2000 matrix, flagging pairs below ~ΔE 5 (and hard-failing < 2). Default the simulation to **deutan + mono** (the common + worst cases).
2. **Warn explicitly when a palette equalizes luminance across hues** — i.e. when chromatic colors meant to be _categorically distinct_ share near-identical L/luminance, they will fail CVD. The tool should detect "these hues differ only in chroma/hue" and surface it.
3. **Distinguish the two palette purposes** (ties to the semantic-ramp work): a **ramp** (one hue, varying lightness — for UI roles) genuinely _wants_ monotone luminance steps; a **categorical set** (many hues at one level — for data/status) _must_ vary in luminance too, or it's CVD-unsafe. Our tool should know which kind it's evaluating and apply the right check. Harmony conflates these: it's a great set of _ramps_ but a CVD-hostile _categorical palette_ if you pick one shade across hues.
4. **Don't trust "perceptually uniform + APCA-tuned" as a proxy for "accessible."** Harmony is both, and still fails CVD. Measured ΔE-under-simulation is the only real check — exactly what the guide says.

**Fair caveat:** Harmony is sold as a _ramp_ palette (Tailwind replacement — you use `red-500` for red things, `green-500` for green things, each as its own scale). Its CVD failure appears specifically when you treat _one shade across many hues_ as a categorical set. That's a real and common use (status colors, charts), and it's the use where our tool can add the most value.

## 6. Reproducibility

- **Data:** `evilmartians/harmony` `source.json` @ `78162e9` (275 DTCG tokens).
- **CVD model:** Machado, Oliveira & Fernandes (2009) severity-1.0 linear-RGB matrices for protan/deutan/tritan; Rec.709 luminance greyscale for monochromacy.
- **ΔE:** colorjs.io `deltaE2000`.
- **Note on simulation:** per the guide's caveat, dichromat-severity simulation gives a _fixed_ worst-ish case; real anomalous trichromats vary. ΔE-on-simulation is a screening tool, not a guarantee — but a palette that fails _this_ certainly fails in practice.
